Leros 采用 Server（控制面）- Worker（执行面） 分离架构，通过 NATS JetStream 作为消息中间件解耦通信。

二、架构层级详解

2.1 进程入口层（cmd）

cmd 进程入口

对应文件: backend/cmd/leros/server.go, backend/cmd/leros/worker.go

进程入口层通过 Cobra CLI 启动两个独立的进程：

  • Server 进程（leros server）—— 控制面，监听 :8080，承载 HTTP API、事件发布、数据库连接。负责接收前端请求，将任务通过 NATS 下发给 Worker。
  • Worker 进程（leros worker [codex|claude|opencode]）—— 执行面，监听 :8081，订阅 NATS 命令执行 Agent Run。通过 NATS JetStream 消费 Server 下发的任务。

该层的职责边界（依据项目约束）：

  • ✅ Cobra 命令注册、进程生命周期管理（lifecycle.Std().WaitExit()）、信号处理、log.Fatal
  • ❌ 任何业务逻辑

Server 与 Worker 是两个独立部署的二进制，通过 NATS 集群通信，天然支持水平扩展 Worker。

2.2 API / 路由层

API HTTP API & 路由

对应文件: backend/internal/api/handler/, backend/internal/api/middleware/, backend/internal/worker/router/router.go

Server 端 API 注册 — api.SetupRouter() 在 Gin 上注册所有 REST 端点：

  • Session 管理（创建 / 列表 / 消息）
  • Project 管理（CRUD + 文件树）
  • Digital Assistant 管理
  • Skill Marketplace（查询 / 安装 / 卸载 / 下载）
  • Connector 管理（GitHub 等外部集成）

Worker 端路由注册 — router.SetupRouter() 注册 Worker 本地 HTTP 端点：

端点作用
GET /health健康检查
/v1/mcpMCP 协议端点（Agent 通过 MCP 调用 Worker 的工具能力）
/v1/chat/completionsOpenAI Chat Completions 模型路由（modelrouter v2）
/v1/messagesAnthropic Messages 模型路由
/v1/responsesOpenAI Responses 模型路由

2.3 Service 层（业务逻辑）

Service 业务逻辑层

对应文件: backend/internal/service/

Server 端的核心业务编排层，处理会话管理、项目操作、Skill 安装/卸载等业务逻辑。该层通过 infra/mq 的 EventBus 向 NATS JetStream 发布 WorkerCommand，将 Agent 执行任务下发给 Worker。

关键类型 WorkerCommand（定义于 backend/pkg/messaging/command.go）：

type WorkerCommand struct { ID string // 消息唯一 ID Type MessageType // "worker.command" CreatedAt time.Time Trace TraceContext // {RequestID, TaskID, RunID} Route RouteContext // {OrgID, WorkerID, SessionID} Body WorkerCommandBody // 根据 CommandType 携带不同 Payload Metadata map[string]any } type WorkerCommandBody struct { CommandType CommandType // agent.run | run.cancel | approval.resolve | question.answer | skill.manage Payload json.RawMessage // 对应类型的 JSON payload ReplyTo string // NATS reply inbox（用于同步回复，如 skill 管理） }

五种命令类型分别路由到不同的 NATS subject（Lane）：

CommandTypeLaneSubject 模式说明
agent.runcmd.runorg.{org_id}.worker.{worker_id}.cmd.runAgent 执行任务
run.cancelcmd.controlorg.{org_id}.worker.{worker_id}.cmd.control取消正在运行的任务
approval.resolvecmd.interactionorg.{org_id}.worker.{worker_id}.cmd.interaction审批决策回传
question.answercmd.interaction（同上）问题答案回传
skill.managecmd.skillorg.{org_id}.worker.{worker_id}.cmd.skillSkill 安装/列表/卸载

2.4 消息中间件层（infra/mq）

infra/mq NATS JetStream 事件总线

对应文件: backend/internal/infra/mq/

基于 NATS JetStream 实现 EventBus 接口，提供持久化消息队列：

  • Publish — Server 向 NATS subject 发布 WorkerCommand
  • Subscribe — Worker 通过持久化 Consumer 订阅对应 subject，保证消息不丢失
  • Request/Reply — 同步请求-响应模式，用于 Skill 管理等需要同步返回的场景（通过 core NATS inbox 机制）

Worker 端的 NATS 连接同时用作 JetStream Consumer（任务消费）和 core NATS（reply 回复）。Server 端用作 Publisher 和 reply Subscriber。

2.5 Worker 命令分发层（command）

command 统一命令分发器

对应文件: backend/internal/worker/command/dispatcher.go

Dispatcher 是 Worker 端统一的 NATS 命令入口，负责四条 lane 的订阅、WorkerCommand 解析和 handler 分发。任一 lane 的订阅异常退出时，Dispatcher 会取消其他 lane 并向进程入口返回错误：

  • Lane 机制：启动后四个独立 goroutine 分别订阅四个 NATS subject lane，反序列化为统一的 WorkerCommand，根据 command_type 分发到对应 handler。
LaneHandler 接口实现者处理逻辑
cmd.run RunHandler command/run.Handler 解码 RunCommandPayload → 校验 → 防抖合并 → WorkerPool 执行 → Seq 追踪
cmd.control ControlHandler command/run.Handler 取消正在运行的 Agent Run（通过 context.CancelFunc）
cmd.interaction InteractionHandler command/interaction.Handler 审批决策 / 问题答案 → 调用 engines.InteractionRouter 唤起阻塞的 Agent
cmd.skill SkillHandler command/skill.Handler Skill install/list/detail/uninstall/import → 通过 core NATS reply 返回结果

Dispatcher 接收原始 *nats.Msg，使 handler 能获取 stream seq 用于 crash recovery（seq tracking）和 背压控制（NATS ACK）。

2.6 Run 命令处理与执行层（command/run）

command/run Run Handler

对应文件: backend/internal/worker/command/run/handler.go, mapper.go, stream_sink.go

run.Handler 同时实现 RunHandler 和 ControlHandler。它处理 cmd.run 与 cmd.control，当前仍承载命令适配、调度和执行准备等多类职责：

执行管道：

Step 1 — Semaphore 获取：背压控制，超过 capacity 时阻塞
Step 2 — Route 校验：验证命令的 OrgID / WorkerID 与本 Worker 匹配
Step 3 — Seq Tracking：通过 SQLite 持久化 NATS stream sequence，实现 crash recovery。Worker 重启后回放时自动跳过已处理的消息。
Step 4 — 防抖合并（Debounce）：同一 Session 的快速连续消息（如用户连续打字）在 1500ms 窗口内合并为一次 Agent Run，减少 LLM 调用次数。
Step 5 — Worker Pool 调度：合并后的任务通过 WorkerPool（默认 20 并发）提交执行，先到先得。
Step 6 — Workspace 准备：从 Gitea clone 项目仓库到临时 Workspace，注入上传附件。
Step 7 — Agent Run：调用 agent.Runner.Run()
Step 8 — 结果流式回传：通过 MQStreamSink 将 Agent 执行的 runtime events 实时发布到 NATS（SSE 推送给前端）

任务取消机制（cmd.control）：

Handler 维护 activeRuns map，记录每个 Session 当前运行的 context.CancelFunc。收到 run.cancel 命令时直接触发 cancel，Agent Run 的 context 即时终止。终端事件使用脱离原取消信号且带超时的发布 context，保证取消后仍能发布 run.cancelled。

2.7 Agent 抽象层（agent/）

agent Agent 抽象与路由

对应文件: backend/internal/agent/runner.go, backend/internal/agent/router.go

定义了核心抽象 Runner 接口：

type Runner interface { Run(ctx context.Context, req *RequestContext) (*RunResult, error) }

RequestContext 是归一化的执行快照，包含 Assistant、Actor、Input、Runtime、Model、Policy 等上下文。使得不同 Runtime（Leros native / Claude / Codex / OpenCode）都能消费相同格式的请求。

RuntimeRouter 实现多 Runtime 路由：

  • 注册多个 Runner（按 kind 标识：leros、codex、claude、opencode）
  • 根据请求中 Runtime.Kind 选择对应 Runner
  • 未指定时使用配置的默认 Runtime

RunResult 包含执行结果：RunID、Status（completed/failed/cancelled）、Message、Error、Usage（token 统计）、ToolCalls 记录。

2.8 Runtime 引擎层（runtime/ + engines/）

runtime Runtime 执行引擎

对应文件: backend/internal/runtime/, backend/engines/

Runtime Service（runtime/service.go）：

  • 创建 tools.Registry 注册所有可用工具（Skill Use / Skill Manage / Memory / Todo / Node）
  • 通过 engines/builtin 构建引擎注册表（native + CLI 引擎）
  • 为每个引擎创建 externalcli.Runner，注入审批/提问处理器和 MCP 配置
  • 返回包装了 lifecycle.Runner 的最终 Runner

lifecycle.Runner 包装层：

  • 在原始 Runner 外层注入执行生命周期步骤：Todo 初始化、审批拦截、Workspace artifact 记录
  • 确保每个 Agent Run 都有完整的生命周期管理（启动 → 执行 → 收尾）

externalcli.Runner：

  • 将 RequestContext 转换为对应 CLI 引擎的命令参数
  • 启动外部 CLI 进程（如 claude、codex）执行 Agent 任务
  • 注入 MCP 服务配置、审批/提问处理器
  • 捕获 CLI 的输出和事件，转换为标准 RunResult

engines/ 层：

  • engines/builtin/ — native 引擎与 CLI 引擎的统一注册
  • engines/native/、engines/claude/、engines/codex/、engines/opencode/ — 具体引擎适配器
  • InteractionRouter — 审批决议路由（Agent 等待审批时，前端回传决议后通过此路由唤醒阻塞的 Agent）

2.9 Run Event 投影层

projector Server 端状态与流投影

对应文件: backend/internal/runnable/session_run_state_projector.go, session_run_stream_projector.go

Worker 通过 MQStreamSink 将标准 runtime event 转为 messaging.RunEvent，再按事件语义发布到两条 lane：

  • run.stream：message、reasoning、tool、todo 等高频增量事件，供 SSE 实时推送和回放。
  • run.state：run started/terminal、artifact、approval、question 等低频关键状态。

Server 分别启动 stream projector 与 state projector。两条 lane 的 JetStream sequence 独立记录：StreamStartSeq 只能来自首条 run.stream，StateStartSeq 来自 run.started。

2.10 Worker 实例调度层（scheduler/）

scheduler Worker 实例调度

对应文件: backend/internal/worker/scheduler.go, backend/internal/worker/scheduler/

定义 WorkerScheduler 接口，管理 Worker 进程/容器的生命周期：

type WorkerScheduler interface { Start(ctx context.Context, spec *WorkerSpec) (*WorkerInstance, error) Stop(ctx context.Context, workerID string) error Health(ctx context.Context, workerID string) error List(ctx context.Context) ([]*WorkerInstance, error) }

三种实现（通过 factory 按配置选择）：

SchedulerEnvType说明
ProcessSchedulerprocess本地进程模式：fork leros worker 子进程，管理 PID 生命周期
DockerCLISchedulerdocker-cliDocker 容器模式：通过 docker run 启动 Worker 容器
KubernetesSchedulerkubevirtK8s 模式：通过 Kubernetes API 创建 Pod 运行 Worker

Worker Server（server/server.go）通过 WebSocket 管理 Worker 的连接、心跳、注销，并暴露 CreateWorker API 供外部动态创建 Worker 实例。

3.2 逐步详解

Step 1 — 前端请求到达

用户发送消息 → POST /v1/sessions/:id/messages → api/handler/ 中的 Session Handler 处理。

Step 2 — Server 构建 WorkerCommand

Service 层调用 messaging.NewRunCommand() 构建 WorkerCommand，包含完整上下文：Assistant 配置、Model 参数、Actor 身份、Workspace 信息、Input 消息列表。

Step 3 — NATS 发布

Server 通过 EventBus.Publish() 将 JSON 序列化的命令发布到 org.{org_id}.worker.{worker_id}.cmd.run subject，JetStream 持久化存储。

Step 4 — Worker 消费

Worker 的 command.Dispatcher 为每条 lane 启动独立 goroutine，通过 JetStream Consumer 拉取消息，解析为 WorkerCommand，并路由到 command/run.Handler。

Step 5 — Run Handler 执行管道

run.Handler.HandleRunCommand() 执行完整管道：Semaphore 获取 → Payload/Route/Model 校验 → Seq 去重 → 防抖合并 → WorkerPool 调度 → Workspace 准备（Clone Git repo、下载附件并 commit） → 调用 agent.Runner.Run()。

Step 6 — Agent 执行与结果回传

lifecycle.Runner 先创建 RunJournal 并执行生命周期 pipeline，再由 RuntimeRouter 根据 kind 选择 externalcli.Runner。引擎事件经 RunJournal 统一编号和归档后，由 MQStreamSink 转为 messaging.RunEvent，分别发布到 run.stream 或 run.state。Server projector 持久化关键状态，并向前端提供 SSE 实时流与回放。

四、关键设计决策

设计点决策原因
消息协议 统一 WorkerCommand envelope 所有 Server→Worker 通信使用同一种消息格式，通过 CommandType 区分类型，避免多套协议维护成本
Lane 分发 四条独立 subject，分别 goroutine 消费 不同命令类型隔离消费，互不影响；高优先级的 control（cancel）不会被 run 的队列阻塞
防抖合并 Session 级别 1500ms 窗口 用户连续打字会产生多条消息，防抖合并后只触发一次 Agent Run，减少 LLM API 调用成本
Seq Tracking SQLite 持久化 NATS stream seq Worker crash 重启后不会重复执行已处理的任务；提升系统可靠性
结果回传 RunJournal → MQStreamSink → 双 lane NATS RunJournal 统一编号和归档；stream/state 分流兼顾实时推送、关键状态持久化和独立 replay sequence
多 Runtime RuntimeRouter + externalcli.Runner 支持 Leros native / Claude / Codex / OpenCode 多种 Agent 引擎，通过 CLI 子进程调用，易于扩展新引擎
审批/提问 InteractionRouter + channel 阻塞 Agent 遇到审批时阻塞等待 channel；用户操作通过 NATS 回传后关闭 channel 继续执行
Worker 调度 Process / Docker / K8s 三模式 适应不同部署环境：开发用 Process，生产用 Docker 或 K8s

六、架构优化项

当前消息协议和双 lane 投影已经完成统一，但 Agent Run 的 Worker 内部边界仍存在以下问题。详细目标架构与迁移步骤见 AGENT_RUNTIME_ARCHITECTURE_PLAN.md。

P1 Runner 接口跨越三层语义

agent.Runner 同时被 RuntimeRouter、lifecycle.Runner 和 externalcli.Runner 实现。调用签名相同，但分别代表 Runtime 选择、Run 生命周期编排和 Engine 驱动适配，导致调用链难以判断真正的职责所有者。

调整方向：顶层使用 RunService 表达一次 Run 的编排；下层使用 Runtime 表达具体执行能力，不再复用同一个泛化 Runner 接口。

P1 run.Handler 职责过多

command/run.Handler 当前同时负责 wire 解码、路由校验、seq tracking、debounce、并发控制、Workspace/附件准备、活动 Run 管理、取消、Agent 调用和失败事件补发。

调整方向：Handler 保留 NATS 入站适配；调度、合并和取消收敛到 RunCoordinator；Workspace 与生命周期收敛到 Agent Run 编排器。

P1 终端事件存在多个潜在生产者

Engine event、externalcli.Runner、lifecycle journal 和 run Handler 都会参与 completed/failed/cancelled 的识别或补发。虽然当前 RunJournal 已统一编号与归档，但终端事件所有权仍不够单一。

调整方向：Runtime 只输出活动事件和执行结果；RunService 是 started 与 terminal event 的唯一生产者；NATS Publisher 只负责 wire 映射和 lane 路由。

P2 wire、domain 与 delivery metadata 混在 runTask

messaging.RunCommandPayload 是跨进程协议，agent.RequestContext 是内部执行快照，二者应保持独立。当前额外的扁平 runTask 又混入了 NATS sequence 等 delivery metadata，形成三套相近结构。

调整方向：保留 wire/domain 分离和显式 mapper；删除中间 runTask，由 RunSubmission 组合 Agent Request、reply message IDs 与 delivery sequences。禁止让 pkg/messaging 依赖 internal/agent。

P2 并发和等待状态分散

Semaphore、WorkerPool、debouncer、pending waiters 和 activeRuns 分别维护部分调度状态，增加取消、关闭和 ACK 时序的推理成本。

调整方向：统一由 RunCoordinator 管理并发、debounce、批次完成通知和活动 Run；NATS Handler 只在 Coordinator 返回后完成 delivery 状态更新。

七、当前结论

范围当前状态后续原则
Server / Worker 通信pkg/messaging 统一 envelope 与 subject保持现有 wire schema，不重新设计消息总线
Worker 命令入口command.Dispatcher 统一四条 lane保持 Dispatcher，继续压薄具体 Handler
