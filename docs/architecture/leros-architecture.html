Leros Server / Worker / Agent Runtime Architecture

重构后的服务端边界：Server 负责命令与投影，Worker 负责业务 Run 生命周期，backend/agent 是独立执行内核；本文同时说明整体、Server、Worker、CLI 与目录层级关系。

Agent Runtime 边界

  • • `backend/agent` 只定义执行契约、Runtime、Tool、Interaction 和 NodeEvent。
  • • native / claude / codex / opencode 是同级 Runtime Adapter。
  • • Runtime 不发布 NATS、不分配业务 Seq、不上传 Plan、不访问业务数据库。
  • • CLI 公共逻辑收敛在 `agent/runtime/internal/{cli,process,todo}`。

Worker Run 生命周期

  • • `worker/command/run` 将 WorkerCommand 转成 RunRequest。
  • • `worker/run.Coordinator` 处理并发、合并、排队和取消。
  • • `worker/agentrun` 独占 Preparer、NodeHandler、Finalizer、Journal 和终态。
  • • `worker/eventpub` 只发布已经构造完成的 `messaging.RunEvent`。

Server ↔ Worker 协议

  • • Server 解析模型配置和 API Key 后发送 WorkerCommand。
  • • Worker 通过 ExecutionRequest 透传执行依赖，不查询业务数据库。
  • • NodeEvent 由 Worker 选择性映射为 RunEvent，Server/UI 只消费 RunEvent。
  • • Live Projector 与 Replay Projector 共享同一套业务事件语义。
01

架构设计思想

Leros 服务端采用“契约驱动 + Server/Worker 分离 + 独立 Agent Runtime”的结构。核心目标不是堆更多框架层，而是把命令、业务生命周期、执行内核和投影语义拆到清楚的位置。

Contract-driven Service Architecture

系统能力先通过明确契约表达，再由 service、worker host 或 runtime adapter 执行。HTTP、NATS、CLI 都只是输入输出适配层，不把业务语义藏在 transport 里。

关键契约

WorkerCommand RunEvent ExecutionRequest ExecutionResult NodeEvent

Runtime First, Worker as Host

backend/agent 只关心“如何执行”。它可以独立测试、替换和扩展。Worker 才关心“这次执行对 SingerOS 业务意味着什么”，包括 Run 终态、Plan 发布、Session 绑定和事件归档。

边界判断

  • • 描述执行事实：放入 backend/agent。
  • • 描述业务 Run 生命周期：放入 backend/internal/worker/agentrun。
  • • 描述面向 UI 的状态：放入 Server service / projector。

Event Ownership

事件不是一层层随手转发，而是每层拥有自己的语义。Runtime 只发 NodeEvent，Worker 映射出 RunEvent，Server 再投影成 SessionEvent / 持久状态。

事件语义

  • • NodeEvent：Runtime 执行节点事实。
  • • RunEvent：Worker 对业务 Run 的统一事实。
  • • SessionEvent：Server 面向 UI 的展示视图。

依赖方向可审计

代码依赖必须朝稳定内核和契约收敛。Agent 不反向依赖业务包，EventPub 不理解 NodeEvent，cmd 不承载业务逻辑。这让架构测试可以用 import guard 保护边界。

禁止穿透

  • • backend/agent 不依赖 backend/internal、config、pkg/messaging。
  • • worker/eventpub 不依赖 Agent 或 AgentRun domain。
  • • cmd/leros 只做进程生命周期和依赖装配。

项目架构层级目录

目录组织按职责而不是技术名词切分。下图是服务端开发时最重要的目录地图：先判断改动属于入口、Server 控制面、Worker 执行面、Agent 执行内核，还是共享契约。

backend/ ├── cmd/leros/ # Cobra CLI, process lifecycle, config bootstrap ├── internal/ │ ├── api/ # HTTP adapters, DTO, route handlers, SSE │ ├── service/ # Server business services and command creation │ ├── runnable/ # Server replay/state projectors │ ├── infra/ # DB, NATS, filestore, external infrastructure │ └── worker/ │ ├── app/ # Worker composition root │ ├── command/ # WorkerCommand adapters │ ├── run/ # Coordinator: debounce, concurrency, cancel │ ├── agentrun/ # Preparer, NodeHandler, Finalizer, Journal │ ├── eventpub/ # complete RunEvent → NATS publish │ ├── runtimehost/ # Worker-side runtime ports │ ├── mcp/ # Worker MCP tool gateway │ └── router/ # Worker local HTTP routes ├── agent/ │ ├── executor.go # Runtime invocation and SerialObserver │ ├── runtime.go # ExecutionRequest / Runtime │ ├── node_event.go # strong typed NodeEvent │ └── runtime/ │ ├── native/ │ ├── claude/ │ ├── codex/ │ ├── opencode/ │ └── internal/ # cli, process, todo shared runtime facilities ├── pkg/messaging/ # WorkerCommand / RunEvent wire contract ├── tools/ # business tool implementations ├── skills/ # skill resources ├── types/ # shared domain types └── config/ # configuration structs

层级说明

  • • cmd/leros：唯一允许处理 Cobra、信号、进程退出和 log.Fatal 的层。
  • • internal/api：把 HTTP/SSE 请求转换为 service 调用，不写业务执行逻辑。
  • • internal/service：Server 侧业务服务，创建命令、管理持久状态、投影 RunEvent。
  • • internal/worker：Worker 侧业务执行 Host，不把进程生命周期拿到库代码里。
  • • backend/agent：业务无关执行内核，Runtime Adapter 与执行事件统一契约。
  • • pkg/messaging：Server 与 Worker 的唯一 wire contract，跨进程共享。
经验法则：如果代码需要知道 NATS subject、业务 Run ID 或 Session 投影，它不应该进入 backend/agent；如果代码需要知道 Cobra flag 或 OS signal，它不应该进入 internal/worker。

层级间关系与核心数据流

核心链路由两条方向组成：命令向 Worker 下发，事件向 Server 回流。Agent Runtime 位于 Worker 内部，但通过独立契约隔离业务语义。

HTTP / SSE / connector input → Server service → WorkerCommand → Worker Coordinator → AgentRun → agent.Runtime → NodeEvent → RunEvent → Server projector / UI

04

Server 架构说明

Server 是控制面：接收用户、外部连接器和前端请求；解析业务配置与模型配置；发布 WorkerCommand；消费 RunEvent 并投影成持久状态和 UI 可读事件。

API / Handler

  • • 只做 HTTP 参数解析、鉴权上下文、DTO 转换和 service 调用。
  • • SSE 不理解 Runtime，它只消费 Server 投影后的 SessionEvent。
  • • Connector 事件进入 Server 后也应先转换为业务命令。

Service / Command Source

  • • 负责业务校验、Session/Project 状态读取和模型配置解析。
  • • Server 在发送 WorkerCommand 前准备完整 ModelOptions。
  • • 不直接调用 agent.Executor 或 Runtime Adapter。

Projector / Replay

  • • Live 与 Replay 都消费 messaging.RunEvent。
  • • Projector 负责恢复 Session 状态、消息片段、工具记录和 Plan 产物。
  • • UI contract 与 Worker wire contract 解耦。
05

Worker 架构说明

Worker 是执行面和业务 Run Host。它不拥有业务数据库，不负责 Server 投影；它的核心价值是把命令变成一次可取消、可归档、可恢复语义清晰的 Agent 执行。

Command & Coordinator

  • • Command handler 只做 wire contract 解析和 RunRequest 构造。
  • • Coordinator 是 Worker 本地调度器，不理解 Runtime Provider 细节。
  • • cancel、debounce、并发限制和 close 都在这一层处理。

AgentRun Host

  • • Preparer 构建不可变 ExecutionRequest。
  • • NodeHandler 选择哪些 NodeEvent 能外发为 RunEvent。
  • • Finalizer 生成唯一业务终态，Journal 分配 Seq 并保证发布顺序。

RuntimeHost & Ports

  • • Provider SessionStore、CLI 路径、Worker 本地目录属于 Worker runtimehost。
  • • OpenCode plan.ready 由 Worker 校验、读取、上传并转成 plan.published。
  • • Worker 透传 API Key 给 Runtime 使用，但不把它写入事件、日志和结果。
06

CLI / 进程入口架构说明

backend/cmd/leros 是进程入口层，而不是业务层。它负责 Cobra 命令、配置加载、依赖装配、生命周期和退出处理；一旦进入业务库代码，就不再处理信号、os.Exit 或 log.Fatal。

Server Command

  • • 装配 DB、NATS、HTTP router、Connector、Projector 和服务依赖。
  • • 启动控制面 HTTP/SSE 能力。
  • • 负责进程级错误处理和优雅退出。

Worker Command

  • • 装配 Worker app、Runtime registry、MCP、modelrouter、NATS subscription。
  • • 把静态配置解析成值传给库代码，不使用运行时全局 setter。
  • • 不在 cmd 层写 Run 编排逻辑。

Local CLI Commands

  • • chat、session、project、skill、login/logout 等命令面向本地开发与操作。
  • • 它们可以调用 API 或本地 helper，但不绕过服务端业务边界。
  • • 新命令先参考 backend/cmd/leros 现有命令骨架。
