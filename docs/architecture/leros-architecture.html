Lework Server / Worker 与 Agent 执行架构设计说明

本文档描述 Server（控制面）、Worker（执行面）以及独立 Agent 执行层的当前职责边界。2026-07-01 完成的 cmd.run 异步分发改造引入了 durable inbox 和 at-least-once 崩溃恢复机制。所有描述均以当前代码为准。

文档状态：当前实施架构。最近变更：cmd.run lane 采用 SubscribeManualDurable + SQLite inbox 异步分发，替代了之前的同步 SeqTracker 方案。

Lework 采用 Server（控制面）- Worker（执行面） 分离架构，通过 NATS JetStream 解耦通信。Worker 内部进一步分为：通信订阅与分发、消息确认控制、持久化 inbox、调度协调、Lework 业务包装和独立 Agent 执行 六层。

二、架构层级详解

2.1 进程入口层（cmd）

cmd 进程入口

对应文件: backend/cmd/lework/server.go, backend/cmd/lework/worker.go

进程入口层通过 Cobra CLI 启动两个独立的进程：

  • Server 进程（lework server）—— 控制面，监听 :8080，承载 HTTP API、事件发布、数据库连接。
  • Worker 进程（lework worker [codex|claude|opencode]）—— 执行面，监听 :8081，通过 NATS JetStream 消费 Server 下发的命令。

Worker 进程在启动时执行以下顺序：

  1. 加载并校验配置（org_id / worker_id 必须）
  2. Bootstrap 引擎（同步内置 skill、配置 CLI MCP）
  3. 创建 NATS 连接、Memory Store、Runtime Service
  4. 创建 RunHandler（含 SQLite inbox，使用共享 lework.db）
  5. 调用 RecoverNonTerminal：恢复上次崩溃时未完成的 inbox 记录
  6. 创建 Dispatcher，启动四个 lane 订阅 goroutine
  7. 注册生命周期关闭函数
  8. 等待退出信号

2.4 消息中间件层（infra/mq）

infra/mq NATS JetStream 事件总线

对应文件: backend/internal/infra/mq/

基于 NATS JetStream 实现 EventBus / Publisher / Subscriber 接口。新增以下关键抽象：

ManualDelivery 接口

对应文件: backend/internal/infra/mq/delivery.go, backend/internal/infra/mq/nats_delivery.go

ManualDelivery 抽象了一条需要手动确认的 NATS JetStream 消息。由 cmd.run lane handler 使用，以精确控制 Ack 的调用时机：

type ManualDelivery interface { Metadata() (*Metadata, error) // Stream/Consumer/NumDelivered Ack() error // 确认成功 Nak() error // 请求重试 NakWithDelay(delay time.Duration) error // 延迟重试 Term() error // 永久终止，不再重试 InProgress() error // 延长 Ack 超时 }

生产实现 natsDelivery 包装 *nats.Msg；测试代码可通过 fake 实现验证确认序列。

SubscribeManualDurable

对应文件: backend/internal/infra/mq/bus.go:38

Subscriber 接口新增 SubscribeManualDurable 方法。与 Subscribe（自动 Ack）的区别：

方法Ack 模式使用场景
Subscribehandler 返回后自动 Ack，panic 时自动 Nakcontrol / interaction / skill lane
SubscribeManualDurablehandler 自行调用 Ack/Nak/Termcmd.run lane

2.5 Worker 命令分发层（command）

command 统一命令分发器

对应文件: backend/internal/worker/command/dispatcher.go

Dispatcher 是 Worker 端统一的 NATS 命令入口，负责四条 lane 的订阅、WorkerCommand 解析和 handler 分发。

  • run lane 使用 SubscribeManualDurable：handler 收到原始 *nats.Msg，由 dispatcher 包装为 ManualDelivery 后传入手动确认 handler。
  • 其余 lane 使用 Subscribe（自动确认）：handler 同步完成处理后自动 Ack。
Lane订阅模式Handler 接口实现者处理逻辑
cmd.run 手动确认 RunHandler command/run.Handler 校验 → 准入 sem → SQLite inbox 持久化 → goroutine 异步执行 → Ack；永久错误 Term，临时错误 NakWithDelay

2.6 Worker Run 异步分发层（command/run）

run handler 异步分发 + 持久化 Inbox

对应目录: backend/internal/worker/command/run/

这是 2026-07-01 的新增核心层。Handler 实现了基于持久化 inbox 的异步分发，提供 at-least-once 的崩溃恢复语义。

消息确认决策

场景确认操作原因
Payload/Route/Model 校验失败Term()永久性错误，重试无意义
inbox 写入失败NakWithDelay(5s)临时故障，等待重试
持久化成功 + 启动后台 goroutineAck()消息已安全持久化，异步执行
无 JetStream metadataNakWithDelay(5s)无法追踪 seq，请求重新投递
准入关闭（shutdown 中）NakWithDelay(5s)等待下次投递
semaphore 等待超时NakWithDelay(5s)并发已满且 InProgress 心跳仍在发送

异步分发流程

  1. admit 统一准入：获取 admission semaphore（等待时每 15s 发送 InProgress 心跳防超时）
  2. SQLite 持久化：调用 PutIfAbsent(topic, seq, cmd)，已存在的记录通过幂等判断处理
  3. 状态检查：检查 admission 是否开放、注册 inflight 追踪、增加 WaitGroup 计数
  4. 启动 goroutine：go dispatchAsync()，然后立即 Ack
  5. 异步执行：goroutine 中构建 RunSubmission → Coordinator.Submit → 执行完成 → inbox MarkCompleted / MarkFailed

2.7 持久化 Inbox 层（run/inbox）

SQLiteRunInbox Durable Inbox

对应文件: backend/internal/worker/command/run/inbox/inbox.go

SQLiteRunInbox 使用 SQLite 存储 WorkerCommand JSON，key 为 (topic, stream_seq) 唯一约束。

表结构（worker_run_inbox）：

CREATE TABLE worker_run_inbox ( id INTEGER PRIMARY KEY AUTOINCREMENT, topic TEXT NOT NULL, -- NATS subject stream_seq INTEGER NOT NULL, -- JetStream 序列号 command TEXT NOT NULL, -- WorkerCommand JSON status TEXT NOT NULL DEFAULT 'pending', -- pending / processing / completed / failed error_msg TEXT NOT NULL DEFAULT '', created_at INTEGER NOT NULL DEFAULT (strftime('%s','now')), updated_at INTEGER NOT NULL DEFAULT (strftime('%s','now')), UNIQUE(topic, stream_seq) );

RunInbox 接口：

type RunInbox interface { PutIfAbsent(ctx, topic, streamSeq, cmd) → (inserted, existingRecord, error) MarkProcessing(ctx, topic, streamSeq) → error MarkCompleted(ctx, topic, streamSeq) → error MarkFailed(ctx, topic, streamSeq, errMsg) → error GetNonTerminal(ctx, topic) → records // 崩溃恢复用 DeleteTerminalBefore(ctx, topic, before) → count // 清理过期记录（72h 保留） Close() → error }

Inbox 保留终端记录 72 小时，启动时自动清理过期项。

2.8 Worker Run 调度层（run/）

worker/run RunCoordinator

对应目录: backend/internal/worker/run/

Worker Run 调度层只处理通信与进程内调度，不生成 prompt，也不理解 Model、Skill、Memory、Workspace 或 Runtime 协议。

RunCoordinator 职责：

  • Session 级 1500ms debounce 与消息合并。
  • 并发限制（默认 20）、pending waiter、active run 注册与 graceful close。
  • 通过 context.CancelFunc 处理 cmd.control。
  • 聚合 delivery sequence，但不依赖 NATS subject 或 payload 类型。

Coordinator 由 RunHandler 构造时创建（作为 executeSubmission 的接收方），后者包装 assistant.Service.Run 调用。

2.9 Lework Agent 业务包装层

assistant 业务 Run 编排

对应目录: backend/internal/assistant/

assistant.Service 是一次 Lework 业务 Agent Run 的唯一编排入口：

  • Preparer：读取 Session history、Memory 和 Skill，解析模型、Tool、Workspace 与附件，构造纯执行层 ExecutionRequest。
  • Service：发布 run.started，调用 agent.Executor，分类 completed/failed/cancelled，并保证每次 Run 只有一个终端事件。
  • Finalizer：执行 Workspace reconciliation、Artifact 收集和 Git stage/commit/push。
  • Journal：归档完整 activity event、usage、tool call 和统计信息。

Assistant、Actor、Org、Project、Skill、Memory、Artifact 和 Git 只能存在于本层及其业务依赖中。

2.10 独立 Agent 执行层

backend/agent Executor + Runtime

对应目录: backend/agent/, backend/agent/runtime/

type Runtime interface { Name() string Execute(ctx context.Context, request ExecutionRequest, observer Observer) (ExecutionResult, error) }

Executor：

  • 校验已经准备完成的 ExecutionRequest。
  • 通过 Registry 按 name 解析 Runtime。
  • 统一执行生命周期、取消与 Observer 错误处理。
  • 返回不包含业务 RunStatus 的 ExecutionResult。

四个同级 Runtime：

  • native：内部使用 Eino，但只消费 ExecutionRequest。
  • claude、codex、opencode：各自 CLI 或服务协议，复用 externalcli Driver。

执行边界：

  • Runtime 只能产生 message、reasoning、tool、approval、question 和 provider-session 活动。
  • Runtime 不访问 NATS、数据库，不生成业务 run.* 终态，不执行 Git。
  • Tool 使用具名类型或 json.RawMessage，公共签名禁止 map[string]interface{}。

2.11 Run Event 投影层

projector Server 端状态与流投影

对应文件: backend/internal/runnable/session_run_state_projector.go, session_run_stream_projector.go

Worker 通过 assistant.EventMapper 将 core execution event 转为 messaging.RunEvent，发布到两条 lane：

  • run.stream：message、reasoning、tool、todo 等高频增量事件，供 SSE 实时推送和回放。
  • run.state：run started/terminal、artifact、approval、question 等低频关键状态。

两条 lane 的 JetStream sequence 独立记录。

三、崩溃恢复机制

crash recovery at-least-once 语义

Worker 进程崩溃后，未完成的 cmd.run 消息由 SQLiteRunInbox 持久化保留。重启后通过 RecoverNonTerminal 机制恢复：

恢复流程

 Worker 启动
    │
    ▼
 1. 创建 RunHandler（含 SQLiteRunInbox）
    │
    ▼
 2. RecoverNonTerminal()
    │
    ├── 清理 72h 前的过期终端记录
    │
    ├── GetNonTerminal() → 所有 pending/processing 记录
    │
    ├── 注册所有记录为 inflight owned
    │   （防止 NATS 订阅开始后重复投递）
    │
    └── 启动 recoveryFeeder goroutine
         │
         ▼
         逐个通过 semaphore 调度执行
    │
    ▼
 3. 启动 Dispatcher（NATS 订阅开始）
    │
    ▼
    恢复记录 + 实时消息共享同一套
    并发限流和生命周期管理

幂等保障：

  • PutIfAbsent 通过 INSERT OR IGNORE + UNIQUE(topic, stream_seq) 保证重复消息不创建新记录。
  • 如果记录已存在且非终态，检查 inflight 是否已 owned，防止重复提交。
  • 重启后 RecoverNonTerminal 在 NATS 订阅前执行，确保恢复的消息不会被实时消息插队。

四、优雅关闭流程

shutdown 5 步关闭顺序

Worker 进程收到退出信号后，按以下 5 步有序关闭：

 Step 1: 取消 dispatcher context
         cancel() → 停止 NATS 投递，不再有新的回调进来
    │
    ▼
 Step 2: 停止新任务准入
         StopAdmission() → admissionOpen = false
         → close(admissionStopped)
         → 不再有新的 WaitGroup.Add
    │
    ▼
 Step 3: 等待 dispatcher goroutine 退出
         ←dispatcherDone
         → 确保没有活跃的回调访问 Handler
    │
    ▼
 Step 4: Drain 积压任务（超时 30s）
         Drain(30s) → Wait 后台 goroutine + recoveryFeeder
         ├── 成功：关闭所有资源
         └── 超时：保留非终态 inbox 记录
                     重启后由 RecoverNonTerminal 恢复
    │
    ▼
 Step 5: 关闭资源
         RunHandler.Close() → cancel execCtx
         → Coordinator.Close() → RunInbox.Close()
         → Bus.Close()

关键设计：Drain 超时后不强制终止正在执行的任务。非终端 inbox 记录保留在 SQLite 中，Worker 重启后通过 RecoverNonTerminal 恢复执行。这保证了 at-least-once 语义——任务要么在当前生命周期完成，要么在下一次重启中继续。

5.2 逐步详解

Step 1 — 前端请求到达

用户发送消息 → POST /v1/sessions/:id/messages → api/handler/ 中的 Session Handler 处理。

Step 2 — Server 构建 WorkerCommand

Service 层调用 messaging.NewRunCommand() 构建 WorkerCommand，包含完整上下文：Assistant 配置、Model 参数、Actor 身份、Workspace 信息、Input 消息列表。

Step 3 — NATS 发布

Server 通过 EventBus.Publish() 将 JSON 序列化的命令发布到 org.{org_id}.worker.{worker_id}.cmd.run subject，JetStream 持久化存储。

Step 4 — Worker 消费（手动确认）

Worker 的 command.Dispatcher 通过 SubscribeManualDurable 订阅 cmd.run lane。消息到达后，dispatcher 将 *nats.Msg 包装为 ManualDelivery，传递给 RunHandler。

Step 5 — Worker 准入与持久化

RunHandler 验证 payload/route/model 后：① 获取 admission semaphore（等待时每 15s 发送 InProgress 心跳）；② 调用 PutIfAbsent 将 WorkerCommand JSON 写入 SQLite inbox；③ 注册 inflight 追踪；④ 启动后台 goroutine；⑤ 调用 Ack() 确认消息。

Step 6 — 异步执行

后台 goroutine 构建 RunSubmission 并调用 Coordinator.Submit。Coordinator 负责 Session debounce（1500ms 窗口）、并发限制、active run 注册和取消。然后调用 assistant.Service。

Step 7 — 业务准备与独立执行

assistant.Preparer 读取 Session、Memory、Skill、Model 和 Workspace，生成纯 agent.ExecutionRequest。agent.Executor 选择 Runtime。Runtime 只返回 execution event 和 result。

Step 8 — 业务收尾与结果回传

assistant.Finalizer 处理 Artifact 和 Git，Service 决定唯一业务终态。EventMapper 将 core event 转为 messaging.RunEvent 发布到 run.stream / run.state。Server projector 持久化状态并提供 SSE 实时流。goroutine 完成时更新 inbox 记录为 completed 或 failed。

六、关键设计决策

设计点决策原因
消息协议 统一 WorkerCommand envelope 所有 Server→Worker 通信使用同一种消息格式，通过 CommandType 区分
Lane 分发 四条独立 subject，分别 goroutine 消费 不同命令类型隔离；高优先级 control 不会被 run 队列阻塞
run lane 确认模式 手动确认（SubscribeManualDurable） 消息持久化到本地 inbox 后才 Ack，实现 at-least-once 语义
其余 lane 确认模式 自动确认（Subscribe） control/interaction/skill 同步处理，无需手动控制 Ack 时机
崩溃恢复 SQLite inbox + RecoverNonTerminal Worker crash 后未完成的任务自动恢复，不丢不重
幂等保障 PutIfAbsent + UNIQUE(topic, stream_seq) 重启后重复投递/恢复的消息不会创建重复执行
防抖合并 Session 级别 1500ms 窗口 用户连续打字防抖，减少 LLM API 调用成本
并发控制 Admission semaphore + Coordinator 上限 双层限流：入口 sem 防止 inbox 过度写入，Coordinator 上限控制执行并发
InProgress 心跳 semaphore 等待时每 15s 发送 防止 NATS 因确认超时而重新投递正在等待槽位的消息
结果回传 Core Event → EventMapper → 双 lane NATS Core 与 wire event 明确分离；stream/state 分流兼顾实时推送和状态持久化
多 Runtime 统一 agent.Runtime + Registry native / Claude / Codex / OpenCode 完全同级，同一契约
执行与业务 backend/agent + internal/assistant 执行层不包含 Lework 业务；业务层负责 Session、Memory、Skill、Workspace、Git
Worker 调度 Process / Docker / K8s 三模式 适应不同部署环境
Stream 配置 MaxMsgsPerSubject=10000, DiscardOld 容忍异步处理期间的积压，新命令始终可写入

八、实施状态

完成 异步分发与崩溃恢复

cmd.run lane 已实现 SubscribeManualDurable + ManualDelivery 接口 + SQLite inbox 持久化 + RecoverNonTerminal 崩溃恢复 + 5 步优雅关闭。Inbox 保留 72h 终端记录，启动时自动清理过期项。

完成 主链与调度语义

唯一主链为 command/run.Handler → RunCoordinator → assistant.Service → agent.Executor → Runtime。Coordinator 已实现 debounce 全 waiter、并发上限、Session 串行、取消、Close 和 delivery sequence 完成/失败语义。

完成 独立执行边界

backend/agent 递归禁止依赖 internal/*、config、messaging 和业务 Tool 包。业务状态由 internal/assistant 持有。

完成 Runtime 与事件收敛

native、Claude、Codex、OpenCode 作为具体 Runtime 注册；三个 CLI Runtime 复用 externalcli Driver。主路径只有一套 ExecutionRequest、ExecutionResult 和 agent.Event。

完成 旧层和全局状态清理

backend/engines、internal/runtime、internal/agent、eventengine 和 simplechat 已退出代码树。

维护约束 防止边界回退

新增 Runtime 必须实现同一 contract suite；新增 Tool 的公共输入必须为 json.RawMessage 或具名结构；Runtime 不得发布业务 run.* 终态；外部事件必须转换为 WorkerCommand 后进入唯一主链。
