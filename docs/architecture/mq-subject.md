# AI Agent 系统消息模型设计文档

## 1. 背景与目标

在基于消息驱动架构的 AI Agent 系统中（Server + Worker + Human-in-the-loop），消息不仅承担"通信"的职责，更是**系统语义的载体**。

本设计的目标是：

* 构建一套**稳定、可扩展的消息分类体系**
* 明确不同类型消息的**职责边界**
* 支撑：
  * 多 Agent 执行
  * 流式交互
  * 人机协同（Human-in-the-loop）
  * 后续 Workflow / 编排能力

本设计**不涉及具体命名实现（如 subject 设计）**，仅关注抽象层。


## 2. 总体设计原则

### 2.1 单一职责原则

每一类消息只表达一种语义：

* 触发行为
* 描述过程
* 表达结果
* 请求决策
* 协同信号

避免"一个消息干多件事"。


### 2.2 稳定优先

消息分类属于**系统基础协议层**：

* 一旦确定，不应频繁修改
* 扩展应通过"子类"完成，而不是新增顶层类别


### 2.3 面向执行流建模

所有消息围绕一个核心：

> **Task 执行过程**

消息模型必须完整覆盖：

```text
触发 → 执行 → 中间过程 → 结果 →（可能中断）→ 恢复
```


### 2.4 人机协同一等公民

系统必须原生支持：

* 用户确认
* 权限审批
* 输入补充

即：执行流可被"外部决策"打断和恢复。


## 3. 消息分类总览

系统统一定义五类消息：

```text
1. Invocation（调用）
2. Progress（过程）
3. Result（结果）
4. Interaction（交互）
5. Coordination（协同）
```


## 4. 各类消息定义


### 4.1 Invocation（调用类）

#### 定义

用于**触发系统执行某个动作**的消息。

#### 语义

> "我要你做一件事"

#### 特点

* 所有执行的起点
* 通常由外部（UI / API / Server）发起
* 会生成或关联一个执行单元（Task）

#### 典型场景

* 用户发送输入
* 创建任务
* 调度任务到执行节点
* 控制执行（暂停 / 恢复 / 取消）


### 4.2 Progress（过程类）

#### 定义

用于描述**执行过程中的中间状态或输出**。

#### 语义

> "事情正在发生"

#### 特点

* 高频产生
* 非最终状态
* 不要求强一致（可丢失）
* 主要用于实时展示

#### 典型场景

* 流式文本输出（token / delta）
* 执行日志
* 中间步骤（step）
* 思考过程（如推理链可视化）


### 4.3 Result（结果类）

#### 定义

用于表达**执行完成后的最终状态或产出**。

#### 语义

> "事情已经完成"

#### 特点

* 低频但关键
* 必须可靠传递
* 可持久化 / 可回放
* 作为系统"事实来源"（Source of Truth）

#### 典型场景

* Task 成功 / 失败
* Message 完成生成
* 执行结果（artifact / 数据）


### 4.4 Interaction（交互类）

#### 定义

用于表达**执行过程中需要外部决策（人或系统）参与的交互行为**。

#### 语义

> "我需要你做出决定"

#### 特点

* 会中断或阻塞执行流程
* 必须成对出现（请求 / 响应）
* 具备独立状态机
* 支持超时 / 拒绝 / 多人参与

#### 标准状态

```text
PENDING → APPROVED / REJECTED / TIMEOUT / CANCELLED
```

#### 典型场景

* 权限请求（访问资源）
* 用户确认（是否继续）
* 参数补充（缺失输入）
* 审批流程


### 4.5 Coordination（协同类）

#### 定义

用于描述**系统内部组件或多 Agent 之间的协作行为**。

#### 语义

> "系统内部正在协作"

#### 特点

* 不一定对用户可见
* 不一定影响最终结果
* 但对编排、调试、扩展至关重要

#### 典型场景

* Agent 内部步骤事件
* Agent 之间通信
* 工具 / 技能调用
* Workflow 内部信号


## 5. 五类消息关系


### 5.1 执行主链路

```text
Invocation
   ↓
Progress
   ↓
Result
```


### 5.2 中断与恢复

```text
执行中
   ↓
Interaction（请求）
   ↓
外部决策
   ↓
恢复执行
```


### 5.3 横向协作

```text
Coordination（贯穿整个执行过程）
```


## 6. 角色视角映射


### 6.1 Server（控制面）

#### 关注重点

* 全局状态
* 调度执行
* 人机交互桥接

#### 涉及消息类型

```text
Invocation    （入口 + 调度）
Progress      （转发）
Result        （状态推进）
Interaction   （交互管理）
Coordination  （可选）
```


### 6.2 Worker（执行面）

#### 关注重点

* 执行任务
* 产生输出
* 发起交互请求

#### 涉及消息类型

```text
Invocation    （接收任务）
Progress      （输出过程）
Result        （输出结果）
Interaction   （发起请求 / 等待恢复）
Coordination  （协同执行）
```


## 7. 边界与约束


### 7.1 Interaction ≠ Invocation

* Invocation：系统主动执行
* Interaction：系统暂停，等待外部决策


### 7.2 Progress ≠ Result

* Progress：非最终状态，可丢失
* Result：最终状态，必须可靠


### 7.3 Coordination ≠ Result

* Coordination：过程性信号
* Result：最终事实


### 7.4 Message 不是类别

Message 只是数据载体：

* 可以属于 Progress（流式输出）
* 也可以属于 Result（最终生成）


## 8. 可扩展性设计

该模型支持以下未来能力：

* 多 Agent 协同
* Workflow 编排
* 审批系统
* 自动化工具调用
* 人机混合决策系统

扩展方式：

* 在现有五类下增加子类
* 不新增顶层类别


## 9. 总结

本系统消息模型可以归纳为：

```text
Invocation    → 我要你做
Progress      → 我正在做
Result        → 我做完了
Interaction   → 我要你决定
Coordination  → 我们在协作
```

该五类模型构成了系统的**稳定语义基础层**，为后续：

* 通信协议
* 状态机设计
* 调度系统
* UI 交互

提供统一抽象。
